# Homey App Architecture

SDK: **Homey SDK v3** (`"sdk": 3` in `app.json`). Entry point: `app.js`.

Official reference:
- [Homey Apps Developer Docs](https://apps.developer.homey.app/)
- [SDK v3 API Reference](https://apps-sdk-v3.developer.homey.app/)

## Directory Layout

```

app.json                        # GENERATED by `homey app build` — do not edit directly
api.js                          # Homey app API endpoints (see § App API)

    driver.js                   # Homey.Driver subclass
    device.js                   # Homey.Device subclass
    driver.compose.json         # Driver manifest (name, class, capabilities)
    driver.flow.compose.json    # Driver-scoped flow cards (triggers, actions, conditions)
    driver.settings.compose.json # (optional) Per-device settings schema
    assets/images/              # small.png (75×75), large.png (500×500)

  app-access.js                 # Access flow card registration + login logic

  protectapi.js                 # v1 API (cookie-based)
  webclient.js                  # v1 HTTP client
  websocket.js                  # v1 WebSocket (binary protocol)
  protect-api-v2/
    protect-api.js              # v2 REST: cameras, lights, sensors, chimes, nvr
    web-client.js               # v2 HTTPS client (X-API-KEY header)
    web-socket-events.js        # v2 WS /subscribe/events
    web-socket-devices.js       # v2 WS /subscribe/devices
  access-api-v2/
    access-api.js               # Access REST: doors, readers, hubs
    web-client.js               # Access HTTPS client (Bearer header)
    web-socket.js               # Access WebSocket
    base-class.js               # Base for Access library classes
locales/                        # i18n JSON files (en, nl, de, fr, …)
settings/                       # App settings page (index.html + settings.css)
widgets/                        # Dashboard widgets (see § Widgets)
  camera/
    widget.compose.json         # Widget manifest
    api.js                      # Per-widget API handler
    public/                     # Frontend HTML/CSS/JS
  doorbell/
    widget.compose.json
    api.js
    public/
```

> **Note on `.homeycompose/`:** This is the **source directory** for global flow cards, actions, triggers, and shared capabilities. Edit files here directly — they are committed to source control. The Homey CLI reads `.homeycompose/` when building the app.
>
> **Note on `.homeybuild/`:** This directory is **generated output** produced by `homey app publish`. Never edit files here — they are overwritten on every publish. It is listed in `.gitignore` and excluded from IDE indexing via `tsconfig.json`.

## App Class (`app.js`)

`UniFiProtect extends Homey.App` — the singleton for the entire app.

### SDK v3 `Homey.App` lifecycle hooks

| Hook | When called | Notes |
|------|-------------|-------|
| `async onInit()` | App starts (after install/update/reboot) | Register flow cards, start API connections |
| `async onUninit()` | App is about to stop | Clean up timers, close connections |

Both are inherited from `Homey.App`; override as needed. `onUninit` is often omitted but should close WebSocket connections for clean shutdown.

### This project's responsibilities

- Instantiates `AppProtect`, `AppAccess`, `ProtectAPI v1`, `ProtectAPIV2`, `AccessAPI`
- Listens to `homey.settings.on('set', ...)` → triggers re-login when credentials change
- Exposes `this.homey.app.api` (v1), `this.homey.app.apiV2` (v2), `this.homey.app.accessApi`
- `debug(…)` method: writes to Homey log when `Homey.env.DEBUG === 'true'`, optionally to `/userdata/application-log.log`
- `this.homeyLog = new Log({ homey: this.homey })` — passive crash reporting via `homey-log` (Sentry)

## Dual API Architecture

| Stack | Auth | Base URL | Class |
|-------|------|----------|-------|
| **v1 Legacy** | Cookie + CSRF | `/proxy/protect/api/` | `library/protectapi.js` |
| **v2 Integration** | `X-API-KEY` header | `/proxy/protect/integration/v1/` | `library/protect-api-v2/protect-api.js` |
| **Access v2** | `Bearer` token | Port 12445 | `library/access-api-v2/access-api.js` |

Both v1 and v2 run in parallel. The v2 API key is optional — app degrades gracefully.

## Standard Driver Pattern

```javascript
'use strict';
const Homey = require('homey');

class MyDriver extends Homey.Driver {
    async onInit() {
        this.homey.app.debug('MyDriver initialized');
    }

    onPair(session) {
        const homey = this.homey;
        session.setHandler('validate', async (data) => {
            const nvrip = homey.settings.get('ufp:nvrip');
            return (nvrip ? 'ok' : 'nok');
        });
        session.setHandler('list_devices', async (data) => {
            return (await homey.app.api.getDevices()).map(d => ({
                name: d.name,
                data: { id: String(d.id) },
            }));
        });
    }

    onParseWebsocketMessage(device, payload) {
        if (Object.prototype.hasOwnProperty.call(device, '_events')) {
            if (payload.hasOwnProperty('someField')) {
                device.onSomeChange(payload.someField);
            }
        }
    }

    getUnifiDeviceById(deviceId) {
        try {
            const devices = this.getDevices();
            const device = devices.find(d => String(d.getData().id) === String(deviceId));
            return device || false;
        } catch (e) {
            return false;
        }
    }
}
module.exports = MyDriver;
```

## Standard Device Pattern

### SDK v3 `Homey.Device` lifecycle hooks

| Hook | When called | Signature |
|------|-------------|-----------|
| `async onInit()` | Device starts | — |
| `async onAdded()` | Device first added (after pairing) | — |
| `async onDeleted()` | Device removed by user | — |
| `async onRenamed(name)` | Device renamed in Homey | `name: string` |
| `async onSettings({ oldSettings, newSettings, changedKeys })` | Per-device settings changed | Returns `string` (new title) or `undefined` |
| `async onUninit()` | Device is about to be destroyed | — |

> `onSettings` must throw a human-readable `Error` if the new value is invalid — Homey will show it to the user and revert the change.

```javascript
'use strict';
const Homey = require('homey');

class MyDevice extends Homey.Device {
    async onInit() {
        await this.waitForBootstrap();
    }

    async onAdded() {
        this.log('Device added:', this.getName());
    }

    async onDeleted() {
        this.log('Device deleted:', this.getName());
        // clean up any timers or listeners
    }

    async onRenamed(name) {
        this.log('Device renamed to:', name);
    }

    async onSettings({ oldSettings, newSettings, changedKeys }) {
        this.log('Settings changed:', changedKeys);
        // throw new Error('Invalid value') to reject and revert
    }

    async waitForBootstrap() {
        // For v1: wait until bootstrap is loaded
        if (this.homey.app.api.getLastUpdateId() !== null) {
            await this.initDevice();
        } else {
            this.homey.setTimeout(this.waitForBootstrap.bind(this), 250);
        }
    }

    async initDevice() {
        this.registerCapabilityListener('some_capability', async (value) => {
            return this.homey.app.api.setSomething(this.getData(), value);
        });
        await this._createMissingCapabilities();
        await this._initDeviceData();
    }

    async _createMissingCapabilities() {
        // Add new capabilities, remove deprecated ones
        if (!this.hasCapability('new_cap')) {
            await this.addCapability('new_cap');
        }
    }
}
module.exports = MyDevice;
```

Key rules:
- `waitForBootstrap()` polls every **250 ms** using `homey.setTimeout`
- For **v2-only devices**: poll `homey.app.apiV2.websocket.isWebsocketConnected()` instead
- `getData().id` is the UniFi device UUID — always compare with `String()`
- `_createMissingCapabilities()` handles capability migration between app versions

## Capabilities

### Built-in vs custom capabilities

- **Built-in** Homey capabilities (e.g. `alarm_generic`, `alarm_motion`, `locked`, `onoff`) — reference by ID, no JSON file needed.
- **Custom** capabilities — defined in `.homeycompose/capabilities/<name>.json`. These are **source files**, committed to git, and read by the CLI during build/publish.

### Capability JSON schema

```json
{
  "type": "boolean",
  "title": { "en": "Away mode", "nl": "Wegmodus" },
  "getable": true,
  "setable": false,
  "uiComponent": "sensor",
  "icon": "/assets/away.svg"
}
```

| Field | Values | Notes |
|-------|--------|-------|
| `type` | `boolean`, `number`, `string`, `enum` | Value type |
| `getable` | `true`/`false` | Whether Homey can read the current value |
| `setable` | `true`/`false` | Whether users/flows can set the value |
| `uiComponent` | `sensor`, `toggle`, `slider`, `button`, `picker`, `color` | How shown in device UI |
| `min` / `max` / `step` | number | For `type: number` |
| `units` | `"°C"`, `"%"`, etc. | Displayed next to value |
| `values` | `[{id, title}]` | For `type: enum` |

### Reading and writing capability values in code

```javascript
// Read (sync)
const isAway = this.getCapabilityValue('nvr_away_mode');

// Write + trigger flows (async — await it)
await this.setCapabilityValue('nvr_away_mode', true);

// Register a listener for when a user/flow sets a capability
this.registerCapabilityListener('onoff', async (value) => {
    await this.homey.app.api.setRelayState(this.getData().id, value);
});
```

> **Important:** `setCapabilityValue(id, value)` automatically fires any associated device trigger flow cards. Do **not** fire the trigger card manually after calling it.

### Capability migration

When a new capability is added in an app update, existing devices don't have it yet. Always run migration in `onInit`:

```javascript
async _createMissingCapabilities() {
    if (!this.hasCapability('new_capability')) {
        await this.addCapability('new_capability');
    }
    // Remove deprecated capabilities
    if (this.hasCapability('old_capability')) {
        await this.removeCapability('old_capability');
    }
}
```

## Flow Cards

### Types and locations

| Type | Source location | Fired via |
|------|-----------------|-----------|
| Global triggers | `.homeycompose/flow/triggers/<id>.json` | `homey.flow.getTriggerCard(id).trigger(tokens)` |
| Global actions | `.homeycompose/flow/actions/<id>.json` | `homey.flow.getActionCard(id).registerRunListener(...)` |
| Device triggers | `driver.flow.compose.json → triggers[]` | `homey.flow.getDeviceTriggerCard(id).trigger(device, tokens)` |
| Device actions | `driver.flow.compose.json → actions[]` | `homey.flow.getActionCard(id).registerRunListener(...)` in `driver.onInit()` |
| Device conditions | `driver.flow.compose.json → conditions[]` | `homey.flow.getConditionCard(id).registerRunListener(...)` in `driver.onInit()` |

### Flow card JSON schema example

```json
{
  "id": "ufp_camera_motion_detected",
  "title": { "en": "Motion detected", "nl": "Beweging gedetecteerd" },
  "tokens": [
    { "name": "camera_name", "type": "string", "title": { "en": "Camera name" } },
    { "name": "score", "type": "number", "title": { "en": "Confidence score" } }
  ],
  "args": [
    { "name": "camera", "type": "autocomplete", "placeholder": { "en": "Select camera…" } }
  ]
}
```

### Registering run listeners

```javascript
// Action card
this.homey.flow.getActionCard('ufp_camera_take_snapshot')
    .registerRunListener(async (args, state) => {
        await args.device.takeSnapshot();
        return true; // must return true on success
    });

// Condition card
this.homey.flow.getConditionCard('ufp_nvr_is_away')
    .registerRunListener(async (args, state) => {
        return args.device.getCapabilityValue('nvr_away_mode') === true;
    });

// Device trigger card (fired from device event handler)
await this.homey.flow.getDeviceTriggerCard('ufp_camera_motion_detected')
    .trigger(device, { camera_name: device.getName(), score: 0.95 });
```

### Dynamic autocomplete arguments

When an arg has `"type": "autocomplete"`, register an autocomplete listener:

```javascript
this.homey.flow.getActionCard('ufp_camera_take_snapshot')
    .registerArgumentAutocompleteListener('camera', async (query, args) => {
        const devices = this.homey.drivers.getDriver('protectcamera').getDevices();
        return devices
            .filter(d => d.getName().toLowerCase().includes(query.toLowerCase()))
            .map(d => ({ name: d.getName(), id: d.getData().id }));
    });
```

### Rules

- `tokens[]` array in the JSON **must exactly match** the token object passed to `.trigger(device, tokens)`.
- `runListener` must return `true` (action/condition resolved) or throw a human-readable `Error`.
- Flow card IDs must use the `ufp_` prefix (see [repo-conventions.md](repo-conventions.md)).
- Always register listeners in `driver.onInit()` (or `app.onInit()` for global cards).

## Pair Flow

### Built-in view names

| View | Purpose |
|------|---------|
| `list_devices` | Homey shows the list returned by the `list_devices` handler |
| `add_devices` | Homey adds the devices the user selected |
| `done` | Pairing finished — show success message |
| `login_credentials` | Built-in username/password form (optional) |

Custom views (e.g. `validate`) are HTML files placed in `drivers/<id>/pair/`.

### `session` API in `onPair(session)`

```javascript
onPair(session) {
    // Handle a custom view event
    session.setHandler('validate', async (data) => {
        const nvrip = this.homey.settings.get('ufp:nvrip');
        return nvrip ? 'ok' : 'nok';
    });

    // Return device list for list_devices view
    session.setHandler('list_devices', async () => {
        return (await this.homey.app.api.getDevices()).map(d => ({
            name: d.name,
            data: { id: String(d.id) },
            store: {},          // optional persistent data
            capabilities: [],   // optional capability override
            settings: {},       // optional initial settings
        }));
    });
}
```

| Method | Description |
|--------|-------------|
| `session.setHandler(event, fn)` | Handle a named event emitted from a pair view |
| `session.emit(event, data)` | Send data to the current pair view |
| `session.showView(viewName)` | Navigate to a named view |
| `session.nextView()` | Navigate to the next view in the flow |
| `session.done()` | Finish pairing (equivalent to reaching `done` view) |
| `session.error(message)` | Show error and stay on current view |

The pair HTML page communicates via `Homey.emit(event, data)` and `Homey.on(event, fn)`.

## App API (`api.js`)

The project-root `api.js` exposes HTTP endpoints callable from the settings page (and other apps via Homey's App API).

### How it works

Routes are declared in `app.json → api[]`:
```json
{
  "api": [
    { "method": "GET",  "path": "/status" },
    { "method": "GET",  "path": "/websocketStatus" },
    { "method": "POST", "path": "/testCredentials" }
  ]
}
```

Each named export in `api.js` maps to a route path (camelCase function name → path). The function receives `{ homey, query, body, params }`:

```javascript
// api.js
module.exports = {
    async getStatus({ homey, query }) {
        return homey.app.api.loggedInStatus;
    },
    async testCredentials({ homey, body }) {
        // body contains POST payload
        return { ok: true };
    },
};
```

### Calling from the settings page

```javascript
// settings/index.html (frontend)
const status = await Homey.api('GET', '/status');
const result = await Homey.api('POST', '/testCredentials', { host, port, user, pass });
```

### Rules

- Functions must be `async` and return a JSON-serialisable value.
- Throw an `Error` to return an error response to the caller.
- Access `homey.settings`, `homey.app`, `homey.drivers` etc. via the injected `homey` parameter.
- Never expose secrets — return only what the settings page needs to display.

## Widgets

Dashboard widgets live in `widgets/<id>/` and appear on the Homey desktop.

### File structure

```
widgets/<id>/
  widget.compose.json   # Widget manifest
  api.js                # Widget-specific API handler
  preview-dark.png      # Preview image (dark theme)
  preview-light.png     # Preview image (light theme)
  public/               # Frontend assets (HTML, CSS, JS)
    index.html
```

### `widget.compose.json` schema

```json
{
  "name": { "en": "Camera", "nl": "Camera" },
  "height": 270,
  "settings": [
    {
      "id": "device",
      "type": "autocomplete",
      "title": { "en": "Camera" }
    },
    {
      "id": "refreshInterval",
      "type": "number",
      "value": 10,
      "min": 5,
      "title": { "en": "Refresh interval (seconds)" }
    }
  ],
  "api": {
    "getSnapshotUrl": {
      "method": "GET",
      "path": "/"
    }
  }
}
```

| Field | Notes |
|-------|-------|
| `name` | Localised widget name |
| `height` | Height in pixels (Homey dashboard grid) |
| `settings[]` | User-configurable widget settings |
| `api` | Named endpoints served by `api.js` |

### `api.js` (per-widget)

Each key in the `api` map corresponds to an export in `api.js`:

```javascript
// widgets/camera/api.js
module.exports = {
    async getSnapshotUrl({ homey, query }) {
        const { deviceId } = query;
        const driver = homey.drivers.getDriver('protectcamera');
        const device = driver.getUnifiDeviceById(deviceId);
        if (!device) throw new Error(`Device ${deviceId} not found`);
        return device.cloudUrl ?? 'loading.png';
    },
};
```

### Autocomplete for widget settings

If a setting has `"type": "autocomplete"`, register a listener in `app.js → onInit()`:

```javascript
this.homey.dashboards.getWidget('camera')
    .registerSettingAutocompleteListener('device', async (query, settings) => {
        const devices = this.homey.drivers.getDriver('protectcamera').getDevices();
        return devices
            .filter(d => d.getName().toLowerCase().includes(query.toLowerCase()))
            .map(d => ({ name: d.getName(), id: d.getData().id }));
    });
```

## Settings Storage Keys

| Key | Type | Value |
|-----|------|-------|
| `ufp:nvrip` | string | NVR IP address |
| `ufp:nvrport` | string | NVR port (default 443) |
| `ufp:credentials` | object | `{username, password}` |
| `ufp:tokens` | object | `{accessApiKey, protectV2ApiKey}` |
| `ufp:settings` | object | `{useCameraSnapshot, ignoreEventsNfcFingerprint, ignoreEventsDoorbell, saveLogToPersistentStorage}` |

`homey.settings` is persistent across app restarts and Homey reboots. Read with `homey.settings.get(key)`, write with `homey.settings.set(key, value)`. Listen for changes with `homey.settings.on('set', key, callback)`.
